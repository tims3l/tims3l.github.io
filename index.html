Architecture design

My last major work at ScrollMax Zrt. was the refactoring of a monolithic PHP application. We finally replaced this with ~14 microservice applications (REST API, background services) and ~8 libraries (some examples can be seen in the following card).

The applications communicated with each other via Apache Kafka messages.

In addition to web clients, a desktop (windows) application also used the REST APIs.

Microservices
  • Auth (OAuth2 server); user migration background services, maintenance
  • FileStorage (storing files in the cloud, serving them via HTTP, REST API endpoints, maintenance, etc.).
  • CMS (management of contents, sections, categories, menus)
  • Release (REST endpoints for upload releases, and spreading updates of desktop software versions)
  • Scrollsupport (ticketing system; REST API & background services)
  • Invoicing (create, download, list, search, send invoices, etc.)
  • Eszigno (electronic signature and serve of documents)
  • NAV (automatic, "foolproof" submission of invoices to NAV)
Event driven architecture

Using the event-driven architecture, we used Apache Kafka messages to communicate between the applications.

For example, when a new invoice was created, our invoicing application sent a message to our pdf electronic signature application. It "consumed" the message and authenticated the document.

Then the invoicing application sent a message to the NAV invoicing application, which forwarded the invoice to NAV.

API design

Client applications could request the displayed data via APIs.

All APIs worked on a similar principle, and most of them used the same libraries. The response pattern was similar and easy to read in all applications. The input parameters made it easy to filter, page, and sort the results - the same way in any API.

The development of the APIs was supported by convenient, self-developed PHP libraries.
